# Encoder Output

## Purpose

This document defines how TACHYON should turn rendered frames and audio into final deliverable media.

It exists because output is not a trivial afterthought.
On weak machines, encoding strategy, buffering policy, and I O behavior can materially affect throughput and reliability.

## Why this matters

A rendering engine can have excellent scene evaluation and pixel output but still perform poorly if its output stage is vague.

Common failure modes include:

- writing large temporary frame sequences to slow disks
- stalling the renderer on encoder backpressure
- tying output semantics too tightly to one container or codec
- making deterministic rendering depend on implicit FFmpeg behavior
- forcing unnecessary recompression or intermediate file churn

## Architectural position

The output stage should sit after render planning and frame generation.
It should consume evaluated frames and audio and produce media deliverables through an explicit contract.

The encoder layer is downstream of:

- scene spec
- property evaluation
- render graph
- frame generation
- audio mix or audio passthrough preparation

## Core rule

TACHYON should separate:

- render semantics
- output intent
- encoder backend behavior

The engine should know what it rendered.
The output contract should know what must be delivered.
The encoder backend should know how to write the requested format.

## Output profiles

A render job should not carry raw encoder flags only.
It should be able to target stable output profiles.

Recommended output profile classes:

- delivery
- master
- intermediate
- image_sequence later if needed
- audio_only later if needed

A profile should resolve into explicit codec, container, pixel format, bitrate or quality mode, audio settings, and buffering behavior.

## First delivery targets

The first implementation should keep the matrix intentionally narrow.

Recommended initial targets:

### Delivery
- MP4 with H.264
- MOV with ProRes
- MKV with H.264 or H.265 where appropriate

### Master or intermediate
- MOV with ProRes
- MKV with FFV1 later if intermediate archival output becomes necessary

The first implementation should not chase every codec family at once.
A small reliable set is better than broad weak support.

## Container guidance

Containers should be treated as explicit output choices, not incidental side effects of codec selection.

Initial guidance:

- MP4 for common delivery compatibility
- MOV for ProRes style editorial or master output
- MKV for robust long-running or interruption-tolerant workflows where appropriate

## Codec guidance

The first encoder surface should support a compact set of clearly documented targets.

Initial recommended codecs:

- H.264 for mainstream delivery
- H.265 later where compression efficiency matters and compatibility is acceptable
- ProRes for high-quality master or handoff workflows
- DNx family later if editorial interoperability becomes a real requirement

## FFmpeg integration direction

FFmpeg should be treated as an encoder backend adapter, not as the owner of render semantics.

Recommended direction:

- TACHYON owns timing, frame order, pixel format intent, and audio handoff intent
- FFmpeg performs muxing and final codec execution
- backend invocations should be explicit and reproducible
- no hidden dependence on shell-specific behavior

## Pipe-first strategy

On weak machines, default output should avoid unnecessary image-sequence dumps to disk.

Recommended first strategy:

- render frames into a bounded in-memory queue
- pass frames to an encoder worker
- stream frames into FFmpeg through pipes or equivalent process input
- avoid writing full intermediate frame sequences unless the job explicitly requests it

This should reduce storage pressure and help slow disks from becoming the dominant bottleneck.

## Backpressure policy

The renderer and encoder should not be coupled through an unbounded queue.

Recommended behavior:

- use bounded frame buffering
- expose backpressure explicitly
- allow the renderer to slow down if the encoder cannot keep up
- keep memory budgets configurable
- never let queue growth become accidental implicit caching

## Audio handoff

The output system should support:

- no audio
- passthrough source audio where valid
- rendered or mixed audio generated by the job
- subtitle or caption side outputs later where needed

Audio timing should remain explicit and aligned to the render job contract.

## Pixel format policy

The output layer should document how internal render surfaces map to encoder-facing pixel formats.

This must include:

- alpha preservation policy
- 8-bit versus higher-bit-depth output targets
- color range expectations
- RGB to YUV conversion policy where required
- whether a target is intended as delivery or master output

## Intermediate output policy

Intermediate files should not be mandatory.
They should exist only when the job requests them or when a backend limitation makes them necessary.

If intermediate outputs are used, the system should prefer:

- bounded temporary storage
- explicit cleanup rules
- formats appropriate to the purpose
- no silent downgrade in quality

## Storage-aware behavior

Weak machines often fail on storage and memory pressure before raw compute becomes the main problem.

The output system should therefore support:

- configurable queue sizes
- temporary directory control
- optional chunked output workflows later
- graceful failure when disk space is insufficient
- logging that distinguishes render bottlenecks from encode bottlenecks

## Failure and recovery

The output layer should surface failures clearly:

- encoder startup failure
- codec or container mismatch
- pipe failure
- disk full conditions
- invalid profile requests
- audio and video stream mismatch

The system should make it easy to understand whether a job failed in rendering, encoding, or muxing.

## Determinism rule

Encoding settings should be part of the render job identity whenever they affect the produced output.

Deterministic rendering does not guarantee deterministic deliverables unless output settings are also explicit.

## Recommended first implementation order

1. define output profile schema inside render jobs
2. build a pipe-first FFmpeg adapter
3. add bounded buffering and explicit backpressure
4. support MP4 H.264 and MOV ProRes first
5. add MKV and optional higher-efficiency or intermediate targets later
6. add image sequence export only as an explicit mode

## Non-goals for the first version

- huge codec coverage
- every editorial mezzanine format
- broad hardware-accelerated encode assumptions
- hidden automatic format selection
- mandatory temporary frame dumps

## Guiding principle

TACHYON should treat output as a first-class deterministic subsystem.
On low-end hardware, a disciplined encoder strategy is part of performance architecture, not just final packaging.
